# Primary Spec Violations: Multiple Definitions Sharing One Spec

## Summary

When adding `@[primary_spec]` to the 172 specified definitions in `functions.json`,
we found that 10 definitions map to only 4 unique spec theorems — meaning
**6 definitions share a spec with another definition**.

All 6 violations follow the same pattern: **Rust operator trait impl variants**
(owned vs borrowed, argument order swapped) that Aeneas transpiles into separate
Lean definitions, where one delegates to another.

## The 4 Violation Groups

### 1. EdwardsPoint Add (2 definitions → 1 spec file)

| Definition | Rust signature | Source |
|---|---|---|
| `Shared0EdwardsPoint...Add...add` | `Add<&'a EdwardsPoint, EdwardsPoint> for &EdwardsPoint` | `edwards.rs:L756-L758` |
| `EdwardsPoint...AddEdwardsPoint...add` | `Add<EdwardsPoint, EdwardsPoint> for EdwardsPoint` | `macros.rs:L33-L35` |

The owned variant (from `macros.rs`) simply delegates to the borrowed variant (from `edwards.rs`).
Both are assigned the same `spec_file`: `Specs/Edwards/EdwardsPoint/Add.lean`.

In the Lean file, there are two **different** `theorem add_spec` declarations in
**different namespaces**:
- `curve25519_dalek.Shared0EdwardsPoint...add_spec` — the "real" spec, tagged `@[primary_spec]`
- `curve25519_dalek.edwards.EdwardsPoint...add_spec` — the wrapper spec (just calls the above)

The second theorem has `is_hidden: true` in `functions.json`, meaning it's a
macro-generated wrapper. Yet it still has `specified: true` and a `spec_statement`.

### 2. EdwardsPoint Mul (2 definitions → 1 spec file)

| Definition | Rust signature | Source |
|---|---|---|
| `Shared0EdwardsPoint...Mul...mul` | `Mul<&'a Scalar, EdwardsPoint> for &EdwardsPoint` | `edwards.rs:L855-L857` |
| `Shared0Scalar...Mul...EdwardsPoint...mul` | `Mul<&'a EdwardsPoint, EdwardsPoint> for &Scalar` | `edwards.rs:L867-L869` |

These are the symmetric `Point * Scalar` vs `Scalar * Point` impls. The second
delegates to the first. Both are `is_hidden: false` and both have `specified: true`.

### 3. MontgomeryPoint Mul (4 definitions → 1 spec file)

| Definition | Rust signature | Source | Hidden? |
|---|---|---|---|
| `Shared1MontgomeryPoint...mul` | `Mul<&Scalar, MontgomeryPoint> for &MontgomeryPoint` | `montgomery.rs:L414-L451` | No |
| `Shared1Scalar...mul` | `Mul<&MontgomeryPoint, MontgomeryPoint> for &Scalar` | `montgomery.rs:L463-L465` | No |
| `MontgomeryPoint...MulSharedB...mul` | `Mul<&'b Scalar, MontgomeryPoint> for MontgomeryPoint` | `macros.rs:L93-L95` | **Yes** |
| `Scalar...MulMontgomeryPoint...mul` | `Mul<MontgomeryPoint, MontgomeryPoint> for Scalar` | `macros.rs:L107-L109` | **Yes** |

This is the worst case: 4 Rust trait impls (owned/borrowed × argument order) all
mapping to the same spec file. The two `macros.rs` entries are hidden wrappers.

The delegation chain:
```
Scalar::mul(P) → Shared1Scalar::mul(P) → Shared1MontgomeryPoint::mul(P, scalar)  [the real impl]
MontgomeryPoint::mul(scalar) → Shared1MontgomeryPoint::mul(P, scalar)             [the real impl]
```

### 4. RistrettoPoint Mul (2 definitions → 1 spec file)

| Definition | Rust signature | Source |
|---|---|---|
| `Shared0RistrettoPoint...Mul...mul` | `Mul<&'a Scalar, RistrettoPoint> for &RistrettoPoint` | `ristretto.rs:L969-L971` |
| `Shared0Scalar...Mul...RistrettoPoint...mul` | `Mul<&'a RistrettoPoint, RistrettoPoint> for &Scalar` | `ristretto.rs:L978-L980` |

Same pattern as EdwardsPoint Mul: symmetric `Point * Scalar` vs `Scalar * Point`.

## What's in functions.json?

Each of these definitions has:
- `specified: true`
- A `spec_file` pointing to the same Lean file
- A `spec_statement` with a **different** theorem (different parameter order, different
  Lean function being called) — they just happen to share the short name `mul_spec` or `add_spec`

So `functions.json` records **10 distinct spec_statements** but only **4 spec_files**.
The spec_statements are genuinely different theorems in different Lean namespaces.

## Is there an issue with functions.json?

**Yes, arguably.** The issue is that `functions.json` treats wrapper/delegate
definitions as fully specified when they're really just thin shims. There are two
sub-problems:

### Problem 1: Hidden wrappers have specs

3 of the 10 definitions have `is_hidden: true` (the `macros.rs` wrappers). These are
auto-generated by Rust macros and their Lean translations are trivial one-line
delegations. Yet `functions.json` marks them as `specified: true` with a full
`spec_statement`. If hidden definitions were excluded from spec assignment, the
MontgomeryPoint group would shrink from 4→2 and EdwardsPoint Add from 2→1.

### Problem 2: Symmetric operator impls are semantically duplicates

Even among non-hidden definitions, `Point * Scalar` and `Scalar * Point` are
symmetric wrappers — one always delegates to the other (with argument reordering).
These are distinct Rust trait impls but represent the same mathematical operation.
The spec theorems in the Lean file prove the same postcondition, just with
swapped parameters.

`listfuns` could treat these as a single logical definition (the one that contains
the real implementation) and mark the delegators as not independently specified.

### Recommendation

One approach: `functions.json` should only set `specified: true` for the
**primary implementation** — the definition that does the actual work, not the
wrapper that just calls it. The wrappers could instead have a field like
`delegates_to: "<lean_name of primary>"` to preserve the relationship without
claiming an independent spec.

This would reduce 10 → 4 specified definitions in these groups, restoring the
strict 1-to-1 invariant between definitions and specs.

Alternatively, `@[primary_spec]` on the Lean side naturally handles this: we only
tag the spec for the primary implementation, and the wrapper specs in the same file
don't get tagged. The current state already reflects this — only 4 of the 10
theorems have `@[primary_spec]`, which is correct.
